# Verification Plan Governance 1.76 Plan

## Human Summary

1.76 should add the first Verification And Test Governance layer: Verification
Plan Governance.

The goal is not to make users understand testing, and not to make IntentOS an
automatic test runner. The goal is:

```text
Codex infers what must be verified.
Codex records why those checks are enough.
Codex cannot claim completion when required verification is missing, weak, or
unrelated to the current task.
```

The user-facing experience should stay simple:

```text
This task needs these checks.
These checks passed / failed / were not run.
These gaps block completion / need your decision.
```

The machine-facing rule is:

```text
No task-bound verification obligations, no trusted test evidence.
No trusted test evidence, no verified execution completion.
```

Verification And Test Governance is the larger direction. 1.76.0 is narrower:

```text
Business Rule Closure
  -> Change Impact Coverage
  -> Verification Plan
```

1.76.0 should not implement Test Evidence Report, should not run tests, and
should not change Execution Assurance completion behavior.

## Current Baseline

IntentOS currently has a strong pre-execution and close-out chain:

- Business Rule Closure explains what business behavior must be true.
- Change Impact Coverage maps affected surfaces and can strictly bind to a
  READY Business Rule Closure.
- Execution Assurance binds intent, plan, actual diff, evidence, review, patch
  assessment, and source-system trace before completion can be claimed.
- Unified Closure gives users one final completion answer.

The missing layer is:

```text
Given the business rule and affected surfaces, what verification is required?
```

Without this layer, Codex can still produce incomplete evidence:

- backend validation without frontend or API negative-path checks;
- UI-only checks for a backend rule;
- happy-path-only tests for a rule whose failure path matters;
- broad `npm test passed` evidence that does not prove the current obligation;
- Codex-authored tests that pass because they assert the wrong behavior;
- skipped, flaky, or not-run tests treated as proof.

## Problem Statement

### Problem 1: Test Commands Are Not Test Obligations

Running a command is not enough. A command can pass while the current business
rule remains unverified.

IntentOS needs a layer that answers:

```text
What must be proven for this task?
```

before it accepts:

```text
What command output exists?
```

### Problem 2: Codex Can Write Weak Or Wrong Tests

Tests generated by Codex can be wrong in familiar ways:

- asserting the implementation instead of the intended behavior;
- testing only the happy path;
- using mocks that bypass the real rule;
- checking UI copy but not backend enforcement;
- checking backend enforcement but not user-visible feedback;
- copying the same wrong assumption from the implementation into the test;
- treating snapshot output as correctness;
- marking a skipped or flaky test as passed;
- using stale test output from another task.

1.76 must treat tests as evidence that also needs governance. A test is not
trusted only because it exists.

### Problem 3: Users Should Not Become Test Designers

The user should not need to choose unit, API, UI, E2E, visual, migration,
rollback, or permission tests.

Codex should infer the verification obligations from:

- user intent;
- Business Rule Closure when present;
- Change Impact Coverage;
- project level: BL0 / BL1 / BL2;
- platform profile;
- risk domains;
- execution kind;
- existing project test conventions.

The user should only be asked when a verification choice changes business
behavior, real-world risk, owner responsibility, data handling, release safety,
or cost.

### Problem 4: Existing Projects Have Existing Test Rules

Old projects may already have test commands, CI gates, release smoke checks,
manual QA rules, or platform-specific conventions.

1.76 must not overwrite those rules. It should compare and map them:

```text
IntentOS obligation
  -> existing project test convention
  -> gap / stronger existing rule / conflict / human decision
```

## Goals

- Add Verification Plan Governance as the first read-only Verification And Test
  Governance layer.
- Generate task-bound verification obligations from Business Rule Closure and
  Change Impact Coverage.
- Ensure each affected surface has appropriate verification obligations or a
  concrete not-applicable reason.
- Define how future test evidence must bind to verification obligations.
- Add explicit safeguards for test correctness, including failure-path,
  negative-path, and test-quality obligations.
- Keep ordinary users out of internal testing details.
- Keep existing-project test conventions respected and comparable.
- Prepare a later Execution Assurance integration without changing Execution
  Assurance behavior in the first step.

## Non-Goals

- Do not add an automatic test runner in 1.76.0.
- Do not add Test Evidence Report in 1.76.0.
- Do not make Execution Assurance consume Verification Plan in 1.76.0.
- Do not execute production, provider, payment, migration, DNS, secret, hook,
  CI, deployment, or external-platform actions.
- Do not approve implementation, commit, push, release, or production.
- Do not claim that passing tests proves product correctness or real-world
  business correctness.
- Do not replace Business Rule Closure, Change Impact Coverage, Execution
  Assurance, Release Plan, or existing project test gates.
- Do not force BL2 validation on BL0 / BL1 projects.
- Do not require users to write test strategy documents.

## Position In The Flow

1.76 should place verification planning after impact coverage and before
execution completion:

```text
User request
  -> Business Rule Closure, when needed
  -> Change Impact Coverage
  -> Verification Plan
  -> implementation / docs / adoption / release-prep work
  -> Test Evidence Report
  -> Execution Assurance
  -> Unified Closure
```

1.76.0 should implement the first part only:

```text
Business Rule Closure
  -> Change Impact Coverage
  -> Verification Plan
```

Later patches can add:

```text
Verification Plan
  -> Test Evidence Report
  -> Execution Assurance integration
```

## Core Model

### Verification Plan

Verification Plan is a task-bound, read-only artifact that defines what must be
verified.

It answers:

```text
What evidence would be enough to support completion for this task?
```

It does not answer:

```text
Did the tests pass?
```

It must remain:

- derived from source systems;
- task-bound;
- intent-bound;
- surface-aware;
- risk-aware;
- platform-aware;
- non-authorizing.

It must not become:

- a test runner;
- a release gate owner;
- a replacement for existing project test strategy;
- a black-box approval system.

### Test Evidence Report

Test Evidence Report is a later artifact that records actual verification
results against a saved Verification Plan.

It answers:

```text
Which obligations were verified, with what evidence, in what environment?
```

It must not be implemented in 1.76.0 unless the 1.76.0 plan is explicitly
expanded after review.

### Source Trace

Verification Plan must keep source trace instead of becoming a new authority.

It should include:

```json
"source_systems": [
  {
    "name": "business_rule_closure",
    "status": "RECORDED",
    "ref": "artifact:business-rule-closures/001-service-time.md",
    "source_outcome": "READY_FOR_IMPACT_COVERAGE",
    "digest": "sha256:..."
  },
  {
    "name": "change_impact_coverage",
    "status": "RECORDED",
    "ref": "artifact:change-impact-coverage-reports/001-service-time.md",
    "source_outcome": "CHANGE_IMPACT_RECORDED",
    "digest": "sha256:..."
  }
]
```

`VERIFICATION_PLAN_READY` must not be allowed when source-system refs are stale,
unresolved, unrelated to the current task, or missing required digests.

## Verification Obligation Types

Verification obligations should use a small, stable vocabulary.

Recommended initial types:

| Type | Purpose |
|---|---|
| `UNIT_BEHAVIOR_TEST` | Isolated business or utility behavior |
| `API_POSITIVE_TEST` | Valid API request succeeds as expected |
| `API_NEGATIVE_TEST` | Invalid API request fails with bounded error |
| `UI_INTERACTION_TEST` | User interaction, form state, validation, disabled state |
| `ERROR_COPY_CHECK` | User-facing error or explanation is present and bounded |
| `BACKEND_RULE_TEST` | Server/domain enforcement cannot be bypassed by UI |
| `DATA_MODEL_CHECK` | Persistence, enum, nullability, migration, historical-data behavior |
| `PERMISSION_BOUNDARY_TEST` | Role, tenant, ownership, visibility, or audit boundary |
| `INTEGRATION_CONTRACT_CHECK` | External API, callback, webhook, import/export, sync behavior |
| `REGRESSION_SMOKE` | Existing critical path still works |
| `VISUAL_OR_SCREEN_CHECK` | Screenshot or visual evidence for UI change |
| `MANUAL_VERIFICATION_REQUIRED` | Human or domain owner must verify |
| `RELEASE_SMOKE_CHECK` | Release-readiness smoke, rollback, monitoring, or handoff check |

## Surface Mapping Rules

Verification Plan should consume Change Impact Coverage surfaces. It should not
invent a competing surface model.

Minimum mapping:

| Change Impact Surface | Required Verification Bias |
|---|---|
| `USER_FLOW` | Positive path, failure path, regression smoke |
| `FRONTEND_UI` | UI interaction, error copy, visual/screen check when visible |
| `API_CONTRACT` | API positive and API negative tests |
| `BACKEND_RULE` | Backend/domain rule test; UI-only verification is insufficient |
| `DATA_MODEL` | Data model, migration, historical data, rollback/not-applicable decision |
| `ERROR_COPY` | Error copy check with user-visible evidence |
| `TEST_COVERAGE` | Explicit test evidence obligation, not only broad command output |
| `DOCS_HANDOFF` | Handoff or documentation verification |
| `PERMISSION_RISK` | Role, tenant, ownership, audit, or escalation checks |
| `RELEASE_IMPACT` | Release smoke, rollback, monitoring, and human release-owner boundary |

## Project Level Rules

Verification strength should be calibrated by project level and risk.

| Level | Default expectation |
|---|---|
| `BL0` | Minimal smoke plus explicit not-run and not-applicable reasons |
| `BL1` | Build/lint/typecheck plus surface-mapped unit, API, UI, and regression checks where relevant |
| `BL2` | BL1 plus permission, security, data, rollback, audit, release-readiness, and stronger evidence binding |

Project level does not decide everything. It must be combined with:

- platform profile;
- risk domain;
- customer-data impact;
- permission/security impact;
- release sensitivity;
- existing project test conventions.

## Platform Calibration

Platform profile should affect verification shape, not the governance model.

| Platform | Typical verification forms |
|---|---|
| `web` | lint, typecheck, unit, component, E2E, visual, API |
| `mini-program` | build, page flow, API contract, permissions, developer-tool/manual evidence |
| `ios` | unit, UI test, simulator, archive/build, permission scenarios |
| `android` | unit, instrumented, build variant, permission scenarios |
| `backend` | API, DB, migration, contract, smoke, load where justified |
| `docs-only` | link/checker evidence, no code-test overclaim |

## Test Correctness Safeguards

1.76 must explicitly guard the correctness of tests themselves.

### Rule 1: Tests Must Bind To The Rule, Not The Implementation

Every required obligation must cite at least one source:

- Business Rule Closure ref, when present;
- Change Impact Coverage ref;
- existing project rule, baseline, API doc, SOP, or user decision;
- human/domain-owner decision when required.

Tests that only cite changed files or implementation details are weak evidence.

### Rule 1A: The Plan Must Be Bound To The Current Task

Verification Plan must include:

- `task_ref`;
- `intent_digest`;
- source-system refs;
- source-system digests;
- `verification_plan_digest`.

This prevents mixing one task's Business Rule Closure with another task's Change
Impact Coverage or Verification Plan.

### Rule 2: Required Negative Paths Must Exist

For validation, permission, API, status, data, and business-rule tasks, the plan
must include failure-path obligations.

Examples:

- missing required field;
- invalid enum/status;
- forbidden role;
- cross-tenant access;
- direct API call bypassing UI;
- duplicate submit;
- stale or historical record edit.

Happy-path-only verification is not enough for these tasks.

### Rule 3: UI-Only Or Backend-Only Verification Is Not Enough For Cross-Surface Rules

If Change Impact Coverage includes both UI and backend/API surfaces, the plan
must include obligations for both sides or record a concrete not-applicable
reason.

### Rule 4: Tests Must Be Capable Of Failing For The Right Reason

For implementation tasks, Test Evidence Binding should later require at least
one of:

- negative test result that proves invalid behavior is rejected;
- bad fixture rejected by checker;
- before/after evidence where the old behavior fails and new behavior passes;
- mutation/sentinel evidence for high-risk or BL2 work;
- manual reviewer confirmation that the test asserts behavior, not a mocked
  implementation detail.

1.76.0 should encode this as planning language and checker expectations. It
does not need to execute mutation testing.

### Rule 5: Skipped, Flaky, Or Not-Run Tests Are Not Passed Tests

Verification Plan and future Test Evidence must distinguish:

- `PASSED`
- `FAILED`
- `NOT_RUN_WITH_REASON`
- `SKIPPED_WITH_REASON`
- `FLAKY_REQUIRES_REVIEW`
- `WAIVED_BY_HUMAN_DECISION`
- `NOT_APPLICABLE_WITH_REASON`

Only `PASSED`, justified `NOT_APPLICABLE_WITH_REASON`, and explicitly approved
`WAIVED_BY_HUMAN_DECISION` can support closure. Failed required obligations
block closure.

### Rule 6: Broad Commands Need Task-Specific Meaning

`npm test`, `pnpm test`, `xcodebuild test`, or CI passing is useful but not
sufficient by itself.

The plan must identify which obligation each command is expected to cover.

Broad commands can support regression smoke. They cannot be the only evidence
for a required business-rule, API, backend, data, permission, or high-risk
obligation.

### Rule 7: Generated Tests Need Review Signals

When Codex creates or changes tests, the plan should require review evidence
for test quality when the task is business-critical, high-risk, BL2, or
cross-surface.

Valid review signals can include:

- independent subagent review;
- Review Loop report;
- checker-backed bad fixture;
- human/domain-owner review;
- existing project test convention match.

### Rule 8: Test Data And Environment Must Be Bounded

The plan must not ask Codex to expose secrets, customer data, production data,
or external provider operations.

Environment expectations must be explicit:

- local;
- CI;
- preview;
- staging;
- simulator;
- developer tool;
- manual real-environment observation.

Production observation is never automatically approved by Verification Plan.

## Verification Plan States

| State | Meaning |
|---|---|
| `VERIFICATION_PLAN_READY` | Obligations are specific enough for execution and later evidence binding |
| `NEEDS_BUSINESS_RULE_CLOSURE` | Business-rule task lacks a READY Business Rule Closure |
| `NEEDS_IMPACT_COVERAGE` | Required surface mapping is missing or weak |
| `NEEDS_DOMAIN_OWNER` | Verification depends on domain, finance, tax, HR, legal, compliance, privacy, payment, production, customer-data, or release-owner decision |
| `BLOCKED_BY_UNCLEAR_TEST_SCOPE` | Codex cannot infer what must be verified without user/domain input |
| `BLOCKED_BY_TEST_CORRECTNESS_RISK` | Proposed verification would likely pass without testing the intended behavior |
| `NOT_APPLICABLE_WITH_REASON` | Verification planning is not relevant for the current artifact-only task, with concrete reason |

## Checker Requirements For 1.76.0

`check-verification-plan.mjs` must reject:

1. business-rule tasks without `business_rule_ref`;
2. `business_rule_ref` without matching digest when strict mode is used;
3. missing `impact_ref` / `impact_digest` when required;
4. required Change Impact Coverage surfaces without obligations;
5. `API_CONTRACT` without both positive and negative API obligations;
6. `BACKEND_RULE` verified only by UI obligations;
7. `FRONTEND_UI` verified only by API obligations;
8. `DATA_MODEL` without migration, rollback, historical-data, or concrete
   not-applicable decision;
9. `PERMISSION_RISK` without role, tenant, ownership, audit, or escalation
   verification;
10. BL2 high-risk work with smoke-only obligations;
11. `NOT_APPLICABLE` without concrete reason;
12. claims that tests prove product correctness;
13. claims that Verification Plan approves release, production, deploy, commit,
   push, CI, hook, secret, migration, provider, payment, legal, tax, HR,
   privacy, or compliance actions;
14. manual verification without owner or decision ref when required;
15. test obligations requiring secrets, customer data, production data, or
   external provider action without explicit boundary and human owner;
16. Codex-generated-test scenarios without test-correctness control when the
   task is BL2, high-risk, cross-surface, or business-critical;
17. `VERIFICATION_PLAN_READY` without `verification_plan_digest`;
18. `VERIFICATION_PLAN_READY` without current-task source binding;
19. plan refs or source refs that point to another saved artifact than the
   current report under strict mode;
20. broad test commands used as the only required business-rule, API, backend,
   data, permission, or high-risk verification obligation;
21. bug-fix tasks without a regression obligation;
22. manual verification without owner and decision ref when required;
23. waiver records without owner, decision ref, and blocking impact;
24. docs-only tasks that require heavy E2E testing without a concrete reason.

## Resolver Requirements For 1.76.0

`resolve-verification-plan.mjs` should:

- infer task intent from `--intent`;
- accept `--business-rule-ref`;
- accept `--impact-ref`;
- accept `--project-level BL0|BL1|BL2`;
- accept `--platform web,backend,...`;
- accept `--change-kind BUSINESS_RULE|BUG_FIX|REFACTOR|DEPENDENCY_UPGRADE|CONFIG_CHANGE|DOCS_ONLY|RELEASE_PREPARATION|ADOPTION_MIGRATION`;
- accept `--out <relative-report-path>`;
- emit Markdown with machine-readable JSON evidence;
- produce stable obligation IDs;
- emit `verification_plan_ref` that self-references `--out` when provided;
- emit `verification_plan_digest` computed from the generated plan evidence;
- emit `source_systems[]` for BRC and CIC when available;
- never write target files unless `--out` is explicitly provided;
- never run tests;
- never approve implementation or release.

When no explicit BRC/CIC refs are provided, resolver may create a draft plan
state, but strict checker must not treat it as READY.

`resolve-change-impact-coverage.mjs` should also support `--out
<relative-report-path>` in 1.76.0 so the BRC -> CIC -> Verification Plan chain
uses the same save-and-check pattern. If this is not implemented, the release
must explicitly document the stdout redirection path used to save `impact_ref`.

## Acceptance Criteria

1. `VERIFICATION_PLAN_READY` requires task-bound source refs.
2. Business-rule implementation tasks can require READY Business Rule Closure.
3. Verification Plan can consume Change Impact Coverage surfaces.
4. Each required surface has obligations or a concrete not-applicable reason.
5. API rules require positive and negative obligations.
6. Backend rules cannot be satisfied by UI-only obligations.
7. UI surfaces cannot be satisfied by API-only obligations.
8. Data-model surfaces require migration, rollback, historical-data, or
   concrete not-applicable reasoning.
9. Permission-risk surfaces require role, tenant, ownership, audit, or
   escalation obligations.
10. BL2 / high-risk tasks require stronger evidence than smoke-only checks.
11. The plan records test-correctness controls for Codex-generated or high-risk
   tests.
12. Skipped/flaky/not-run/waived concepts are present for later evidence
   binding, and are not treated as passed tests.
13. Existing-project behavior is read-only and mapping-first.
14. Generated-project smoke proves BRC -> CIC -> Verification Plan binding.
15. Negative fixtures prove the checker rejects weak or misleading test plans.
16. Claim Control rejects any wording that says Verification Plan approves
   implementation, release, production, or product correctness.
17. `VERIFICATION_PLAN_READY` requires `verification_plan_digest`.
18. Verification Plan generated with `--out` self-references its own artifact
   path through `verification_plan_ref`.
19. The resolver must not mark a draft plan READY when BRC or CIC refs are
   missing for a business-rule task.
20. Broad test commands must map to specific obligations or remain non-blocking
   regression smoke only.
21. Task or intent mismatch between BRC, CIC, and Verification Plan must be
   rejected under strict mode.

## Subagent And Review Plan

Use subagents as reviewers, not as uncontrolled writers.

Recommended split:

- one source-reader subagent checks BRC/CIC/Execution Assurance consistency;
- one checker-review subagent reviews failure modes and bad fixtures;
- one docs-review subagent checks ordinary-user language and overclaim risk.

All subagents must be closed after use and their findings summarized into the
main Execution Assurance / release evidence. No subagent may approve release or
production.

## Known Limitations

- 1.76.0 should not execute tests.
- 1.76.0 should not prove that tests pass.
- 1.76.0 should not prove real-world correctness.
- Test Evidence Binding should be a follow-up patch after Verification Plan
  Governance is stable.
- Full mutation testing, flaky-test analytics, and deep static analysis are out
  of scope for 1.76.0.

## Follow-Up Path

Recommended sequence after 1.76.0:

1. 1.76.1 Test Evidence Binding
   - add `test-evidence-reports/`;
   - bind commands, exit codes, output digests, environment, skipped/not-run
     states, and manual checks to Verification Plan obligations.
2. 1.76.2 Execution Assurance Integration
   - make execution-class completion consume Test Evidence where obligations
     exist.
3. 1.76.3 Test Correctness Hardening
   - add deeper cross-checks for Codex-generated tests, failure-path proof,
     mutation/sentinel evidence where justified, and stale-test-output
     rejection.

## Final Position

1.76 should make testing part of IntentOS evidence governance, not a separate
testing framework.

Business Rule Closure says what must be true.
Change Impact Coverage says where it matters.
Verification Plan says what must prove it.
Test Evidence later says what actually proved it.
Execution Assurance decides whether execution can be claimed complete.

The key 1.76 rule is:

```text
Tests are not trusted by default.
Tests become useful evidence only when they are task-bound, behavior-bound,
failure-aware, and checked against the current verification obligations.
```
