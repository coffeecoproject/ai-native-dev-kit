# IntentOS 1.104.1 Release Record

## Release

`1.104.1` - Active Guidance Semantic Hardcut

## Purpose

Make the existing zero-experience solo operating contract executable across
current guidance, prompts, generated Agent/Starter assets, baseline routing,
and public runtime output.

The user communicates business reality. IntentOS/Codex owns Profile,
architecture, stack, baseline, BL level, industrial packs, verification,
review, subagent, hook, checker, workflow, and technical-risk decisions inside
the requested boundary.

## Delivered

- bound current user input to exactly four classes:
  `NO_USER_ACTION`, `BUSINESS_FACT_NEEDED`, `REAL_WORLD_CONSENT_NEEDED`, and
  `EXTERNAL_FACT_NEEDED`;
- expanded semantic enforcement from direct statements to questions, menus,
  tables, ambiguous approval slogans, generic Risk Gate approval, and technical
  decisions inside human-only sections;
- registered current entry guidance, Prompt families, generated Agent assets,
  Starter assets, and baseline/runtime resolvers as active guidance;
- removed Profile, BL, industrial-pack, baseline, test, review, and workflow
  selection menus from current user-facing entry output;
- changed technical uncertainty to stricter internal review, missing-evidence,
  repair, or blocked states rather than asking a zero-experience user to judge
  implementation mechanics;
- preserved exact consent for prepared production, payment, real-user,
  provider, account, cost, and irreversible effects;
- added focused source and generated-project semantic regressions.

## Authority Boundary

This release does not authorize implementation, target-project writes, apply,
commit, push, release, production, payment, migration, provider operations,
secret access, or irreversible data actions. Technical ownership does not turn
an internal recommendation into execution authority.

Compatibility schemas and historical records may retain owner, reviewer,
approval, and human-decision field names. Current consumers must interpret
those fields through the four current user-input classes.

## Evidence Status

The source registry, semantic analyzer, Prompt/Agent/Starter assets, baseline
resolvers, public entry output, and focused regressions provide direct evidence
for the responsibility-model hardcut. Generated-project and release-chain
checks provide distribution evidence. They do not prove business or legal facts.

## Allowed Claims

- Current registered guidance rejects known direct and indirect delegation of
  technical decisions to a zero-experience user.
- Public baseline and project-entry output reports one Codex-selected technical
  route and one of four bounded user-input classes.
- Generated Codex Agent and Starter assets preserve the same responsibility
  model as the source repository.

## Forbidden Claims

- Semantic consistency does not prove business, legal, security, privacy,
  compliance, release, or production correctness.
- Codex technical ownership does not authorize target-project writes, apply,
  commit, push, release, production, payment, migration, or provider actions.
- A general request, silence, internal recommendation, or compatibility approval
  field is not exact real-world consent or external authority.
