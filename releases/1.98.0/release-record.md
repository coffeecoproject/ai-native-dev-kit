# IntentOS 1.98.0 Release Record

## Theme

Execution And Distribution Trust Hardcut.

## Human Summary

IntentOS now makes the systems that already existed mandatory at the places
that claim a task is complete, a workflow is active, a baseline is installed,
or a release can enter review. Ordinary users still use the same `work` entry;
the additional checks run behind that entry.

## Delivered

- complete natural-language routing for start, adopt, discuss, resume, status,
  finish, and release preparation;
- Work Queue conflict and paused-task resume enforcement;
- fail-closed unknown code behavior and open business-rule questions;
- strict Completion Evidence before public completion;
- trusted artifact-schema digests and strict-empty rejection;
- project-local authoritative evidence paths and recursive source binding;
- independent Launch Review and human Release Approval inputs;
- canonical release package identity and trusted requirement matrices;
- project-local plan outputs, full source-state binding, atomic writes, current
  action rollback, and semantic readiness validation;
- Apply Receipt revalidation of installed version, workflow route, read-only
  behavior, baseline structure, and BL2 industrial evidence;
- platform-compatible baseline packs and pending maturity evidence;
- durable new-project origin so managed assets do not reclassify native new
  projects as legacy production projects;
- generated-project release-channel dependency and command parity;
- current 1.x migration routing into controlled update planning;
- independent adversarial tests for empty evidence, schema replacement, plan
  path escape, pack mismatch, Node project classification, release identity,
  generated distribution, and Launch Review separation.

## Allowed Claims

- Strict completion and apply consumers fail closed on missing required input.
- Replaced installed artifact schemas cannot satisfy trusted strict checks.
- Approved controlled apply is invalidated by current target-state drift.
- A verified Apply Receipt is rechecked against current installed behavior.
- Release Approval alone cannot unlock assisted release execution.
- Generated projects include and execute the published release-channel command.

## Forbidden Claims

- Repository tests prove a real production deployment or external provider.
- IntentOS cryptographically proves that a named human approved an action.
- A passing receipt proves product correctness or business correctness.
- Baseline installation certifies security, privacy, compliance, or production.
- IntentOS may deploy, publish, submit, migrate, change secrets, DNS, payment,
  permissions, production configuration, or project authority by itself.

## Known Limitations

- Evidence Authority binds project state and referenced artifacts, but it does
  not prove that an external system, provider, or named human is authentic.
- Controlled apply covers only approved low-risk project-local actions; CI,
  hooks, release systems, runtime configuration, and production remain outside
  automatic apply authority.
- Baseline and release checks prove recorded governance state, not product,
  security, compliance, privacy, or real-production correctness.
- Non-Git projects use a deterministic content revision that excludes generated
  IntentOS workflow-record directories to avoid self-referential evidence;
  every referenced source artifact remains bound by its own raw file digest.
