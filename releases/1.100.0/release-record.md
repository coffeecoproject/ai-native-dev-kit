# IntentOS 1.100.0 Release Record

## Theme

Execution Authority And Consumer Hardcut.

## Human Summary

IntentOS now makes the existing execution rules effective at the final
consumer. A task cannot be called complete because a report merely says it is
complete. A release plan cannot assign provider or production effects to
Codex. A controlled project write cannot proceed through a blocked review and
must restore attempted writes when a later action fails.

The public user model remains one zero-experience solo developer. The user
describes the real business, supplies facts the project cannot know, and gives
consent only to a prepared concrete real-world effect. Codex owns technical
classification, planning, implementation, verification, review, repair, and
evidence handling.

## Delivered

- one normalized release-action authority used by release planning and checks;
- provider, store, production migration, and rollback execution remain human
  or external-system actions even in assisted mode;
- public `finish` consumes Work Queue, Task Governance, conditional Plan
  Review, Execution Assurance, and Completion Evidence;
- material ignored files, untracked files, Git failures, and submodule state
  are represented in project identity;
- controlled apply validates Plan Review before writing and journals the
  current action before mutation so failure rollback covers every attempt;
- existing-project governance and queue discovery use bounded deep scans
  without shallow result truncation;
- BL2 evidence and starter/profile compatibility fail closed;
- natural-language discussion and release preparation routes are separated;
- installed-project CI conditionally enforces current task, completion, and
  release consumers when implementation or release files change.

## Preserved Safety

- no new public workflow mode or user command family;
- no team or enterprise role model;
- no automatic deploy, store submission, production migration, rollback,
  secret, DNS, payment, permission, or production configuration change;
- no replacement of stricter valid project-owned governance;
- no claim that repository checks prove business correctness or production
  safety.

## Evidence Boundary

Repository verification proves the checked source, generated-project assets,
strict negative fixtures, controlled local apply behavior, and distribution
configuration. It does not prove provider account state, app-store review,
real production health, legal authority, or a specific business outcome.

## Evidence Status

The release is checked through focused authority, operating-model, evidence,
apply, adoption, baseline, distribution, and generated-project regressions.
The same source snapshot must also pass strict Manifest validation, repository
self-check, the full verification suite, and the diff check. Provider and
production facts remain external evidence and are not inferred from repository
success.

## Known Limitations

IntentOS can verify project-local plans, identities, digests, receipts, and
consumer wiring, but cannot independently observe provider accounts, store
review state, production traffic, legal permission, or business outcomes.
Installed CI enforces the supported report families; project-specific systems
may add stricter checks. See [known-limitations.md](known-limitations.md).

## Allowed Claims

- External release effects are never assigned to Codex by assisted mode.
- Public completion consumes the current task governance and evidence chain.
- Controlled apply rejects blocked Plan Review and records rollback outcomes.
- Selected baseline and installed-project consumer requirements fail closed.

## Forbidden Claims

- Passing repository checks does not approve production or a store submission.
- Evidence identity does not prove product or business correctness by itself.
- A generated plan, approval record, or readiness report is not write authority
  unless the exact controlled apply chain accepts it.
- Deep discovery does not authorize replacing stricter project-owned rules.
