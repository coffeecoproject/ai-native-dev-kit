# IntentOS 1.99.3 Release Record

## Theme

Review And Execution Trust Close-Out.

## Human Summary

IntentOS now keeps current review, completion, apply, adoption, release, and
installed-project checks on one project-local evidence chain. A current review
cannot be replaced by an old packet, another task, a missing source, a stale
digest, an unsafe path, or an installed project borrowing source-repository
assets.

The public operating model remains one zero-experience solo developer. The user
describes the business and supplies unavailable business facts or consent to a
prepared real-world effect. IntentOS and Codex retain technical planning,
implementation, testing, review, repair, and internal routing.

## Delivered

- current Review Packet and GPT prompt project/task identity binding;
- exact, single-section Review Context Binding validation;
- complete active-guidance registration and installed alias checks;
- project-local Plan Review source resolution and digest verification;
- semantic Plan Review validation in Execution Assurance, Completion Evidence,
  Controlled Apply, Adoption Apply, and Release Evidence consumers;
- exact Review Loop final-state handling in Execution Closure;
- installed manifest and version authority isolation;
- no source-registry borrowing during strict target baseline checks;
- consistent `AGENTS.md`, `agent.md`, and `.agent.md` collaboration entry
  recognition;
- fail-closed starter verification with executed negative-path self-checks;
- current CI schema smoke checks derived from the runtime contract.

## Preserved Safety

- no new user mode, workflow command, evidence family, or organization role;
- no implementation, apply, commit, push, release, or production authority;
- no automatic real-world effect;
- historical evidence remains readable but cannot become current evidence;
- compatibility fields remain machine vocabulary, not people the user must
  appoint.

## Evidence Status

The current source checker validates Review Context classification and binding,
project/task identity, project-local Plan Review sources and digests, strict
consumer propagation, installed manifest isolation, baseline registry
isolation, alternate agent entry files, and fail-closed starter verification.
Repository self-check executes generated-project and installed-project paths.
Final release claims require focused tests, Manifest validation, repository
self-check, the full verification suite, exact CI smoke commands, and the diff
check to pass on the same source snapshot.

## Known Limitations

Deterministic prose checks cover supported direct contradictions but do not
prove every sentence semantically correct. Historical review inputs remain
readable for audit and ready-state checks, but only implementation mode may
count a current project/task/context binding as current implementation
evidence. Tool availability still determines which project-native verification
commands can run. See [known-limitations.md](known-limitations.md).

## Allowed Claims

- Current review inputs identify the current project, revision, and task.
- Plan Review consumers reject unresolved or stale source evidence.
- Installed projects prefer installed manifest, context, baseline, and version
  authority.
- Starter verification reports failure when no real verification path can run.

## Forbidden Claims

- Evidence identity does not prove product correctness by itself.
- A passing Plan Review does not authorize implementation by itself.
- A passing readiness report does not authorize apply or production.
- Deterministic context checks do not understand every possible prose meaning.

## Verification

- focused context, manifest, plan-review, execution, apply, release, baseline,
  starter, and operating-model tests;
- generated and installed project regressions;
- `node scripts/check-manifest.mjs`;
- `node scripts/check-intentos.mjs`;
- `npm run verify`;
- exact PR workflow smoke commands;
- `git diff --check`.

See [known-limitations.md](known-limitations.md) and
[self-check-report.md](self-check-report.md).
