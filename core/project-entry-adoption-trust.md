# Project Entry And Adoption Trust

## Purpose

Project Entry And Adoption Trust is the mandatory internal trust boundary before
IntentOS routes ordinary work, creates a project, assesses an existing project,
or applies adoption assets.

It keeps these dimensions separate:

- target topology;
- current IntentOS identity;
- original goal and execution intent;
- project facts and current work;
- effective Guidance Authority;
- project-native governance authority;
- behavioral adoption;
- release readiness.

No single marker, filename, keyword, or self-declared report can collapse those
dimensions into a stronger state.

## Public Contract

The user provides a natural-language goal. Codex derives topology, project
state, platform posture, baseline, adoption depth, validation, rollback, and
evidence routing.

The user is asked only for a missing business fact, unavailable external fact,
product preference, or exact consent for a prepared real-world effect.
Technical uncertainty remains Codex work.

## New Project Contract

An absent leaf or empty directory may enter controlled setup only when:

1. the complete path topology is safe and unchanged;
2. the original goal is digest-bound;
3. one exact setup action graph is derived;
4. request authority and controlled readiness bind that graph;
5. all writes use one same-filesystem transaction;
6. the generated project cold-starts from its own installed assets;
7. Work Queue and Task Governance route the original goal without writes;
8. a success receipt is persisted before `APPLY_VERIFIED` is returned.

Any failure after the first write restores the original topology or reports an
explicit incomplete rollback. `--force-new-project` is not an authority path.

## Existing Project Contract

A non-empty project is never initialized as a blank project. IntentOS first
collects complete project facts, current work, project-native authority, legacy
workflow evidence, and conflicts. It then derives a behavior-complete adoption
route while preserving stronger project rules.

Read-only migration, reconciliation, convergence, review, and assurance may
exchange one strict same-run evidence chain. That chain is not durable approval
and does not authorize apply.

## Activation Contract

Copied files do not prove activation. `VERIFIED_ACTIVE` requires:

- one exact `APPLY_VERIFIED` receipt;
- a fresh project-local IntentOS cold start;
- a read-only Work Queue and Task Governance route;
- unchanged target and source observations during calibration;
- current project and goal binding.

Activation does not prove product correctness, release readiness, or production
authorization. The first ordinary task is recorded separately and still follows
Business Rule Closure, Business Universe Coverage when required, verification,
review, and Unified Closure.

## Failure Semantics

Identity, Guidance, topology, source-chain, apply, rollback, and activation
failures are fail-closed. Public entries may continue read-only diagnosis, but
must not describe blocked, pending, or rolled-back work as installed, active,
or complete.
