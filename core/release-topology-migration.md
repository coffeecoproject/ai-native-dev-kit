# Release Topology Migration

## Purpose

Release topology migration governs a change between two release execution
topologies without turning IntentOS into a deployment service or production
authority.

The public user is one zero-experience solo developer. Codex owns discovery,
technical comparison, planning, rehearsal design, evidence checks, rollback
design, and the recommended next step. The user supplies only unavailable
business or provider facts and exact consent to one prepared real-world effect.

## Lifecycle

1. `DISCOVERY`: inspect the current six-plane topology without writes.
2. `PLAN`: compare capabilities and record one bounded recommendation.
3. `APPLY_REVIEW`: bind Plan Review, Unified Apply Plan, exact approval,
   readiness, rollback, and later Apply Receipt for project-file changes.
4. `REHEARSAL`: run a non-production shadow path with Verification Runtime
   Trust and exact package identity.
5. `CUTOVER_READINESS`: prove rehearsal, rollback, lock, trigger, executor, and
   evidence conditions.
6. `CUTOVER_REVIEW`: reference separate current-user consent for the exact
   external cutover effect. Cutover consent is not release consent.
7. `POST_CUTOVER_PROOF`: prove the target path ran and the old path did not also
   publish.
8. `RETIREMENT_REVIEW`: separately govern old-backend retirement while
   preserving historical evidence and fallback requirements.

No stage may imply that a later stage is approved or complete.

## Authority

The migration report is a computed read-only projection. It is not an approval
record, Apply Plan, execution receipt, release approval, provider response, or
production observation. Every referenced source remains independently
authoritative and must match the current project, revision, task, candidate,
package, topology, action, and run where applicable.

## Safety Rules

- recommendation is not approval;
- Plan Review is not implementation permission;
- project-file apply is transactional and uses the approved action graph;
- rehearsal is non-production and cannot duplicate deployment or submission;
- embedded consent is never authority;
- cutover and release require separate exact effects when both occur;
- post-cutover proof must be observed, not declared;
- retirement cannot delete historical evidence;
- cleanup is limited to exact run-owned resources;
- stronger project-native rules remain authoritative;
- unknown production effects fail closed.

## User Decision Classes

Only these classes may be exposed:

- `NO_USER_ACTION`
- `BUSINESS_FACT_NEEDED`
- `REAL_WORLD_CONSENT_NEEDED`
- `EXTERNAL_FACT_NEEDED`

The user must not choose an orchestrator, runner, transport, artifact store,
rollback command, lock strategy, or internal IntentOS artifact.
