# Apply Execution Receipt Governance

An Apply Execution Receipt is runtime evidence for one bounded IntentOS
init/update apply. It is generated by the controlled runner after validating an
immutable execution plan, a matching human Approval Record, and a matching
Controlled Apply Readiness Report.

The receipt is evidence, not authority. It does not approve implementation,
release, production, CI, hooks, business code, migrations, secrets, payments,
or changes to project-owned governance.

## Required Trust Chain

```text
execution plan -> approval -> readiness -> replay -> receipt -> activation
```

Every reference must resolve inside the same target project and match the same
plan digest, action IDs, and exact target paths.

## Verified Receipt Rules

`APPLY_VERIFIED` requires:

- only approved plan actions were executed;
- every write has matching before/after evidence;
- source-backed writes match their plan-bound source digest;
- no unexpected target path changed;
- the receipt belongs to the current project root;
- current applied target hashes still match the receipt;
- installed `workflow-next` ran read-only and returned a recognized route;
- rollback information is recorded.

## Failure Rules

The runner stops at the first mismatch. Already-written actions must be rolled
back. A failed or unverified receipt cannot support active adoption.

## Allowed Claims

- The exact approved IntentOS governance graph was replayed.
- Listed target hashes matched after apply.
- No unexpected target path was observed.
- The installed IntentOS workflow entry was reachable.

## Forbidden Claims

- The product is correct.
- Business implementation is approved.
- Release or production is approved.
- Project authority moved to IntentOS.
- Arbitrary project changes may be executed automatically.
