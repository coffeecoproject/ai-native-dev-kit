# Review Context Authority

## Current Authority

The current IntentOS product contract has higher interpretation authority than
compatibility schemas, completed plans, examples, audit logs, and prior release
records.

The default user is one zero-experience solo developer. The user communicates
real business goals, unavailable business facts, product preferences, and
consent to one prepared concrete real-world effect. IntentOS and Codex own all
technical judgment and internal workflow.

Industrial safeguards describe engineering depth. They do not imply an
enterprise audience, a team, departments, or additional people.

## Interpretation Precedence

```text
CURRENT_PRODUCT_CONTRACT
  > CURRENT_RUNTIME_AND_GATES
  > COMPATIBILITY_SCHEMA
  > HISTORICAL_RECORD
```

`CURRENT` assets may define current product behavior. `COMPATIBILITY` assets
may preserve machine readability but must be translated through the current
contract. `HISTORICAL` assets are audit evidence only. `CONFLICTING` active
guidance must fail review-context validation.

## Review Rule

Every reviewer, GPT reviewer, and review subagent must:

1. read the current version and this contract before making product-direction
   recommendations;
2. separate current-task findings from compatibility or historical notes;
3. preserve project-native rules when they are stricter and proven;
4. reject recommendations that add user modes, technical user choices, role
   selection, or automatic capability expansion;
5. keep technical findings detailed while translating the user-facing result
   into business facts, concrete effects, and the next automatic action.

Review must not propose Solo/Team/Enterprise setup modes. It must not infer a
multi-person model from BL2, industrial packs, release safeguards, evidence
chains, or internal responsibility domains. It must not ask the user to select
architecture, stack, database shape, baseline, packs, tests, reviewers,
subagents, hooks, checkers, or workflow commands.

## Compatibility Translation

Active machine fields such as `owner`, `reviewer`, `human_decision`,
`release_owner_ref`, and `cost_owner_ref` may remain for compatibility. They do
not instruct the user to assemble a team. Their current public meaning must be
one of:

- current-user consent to one exact prepared real-world effect;
- an authoritative external fact or provider/account capability;
- an internal responsibility domain used to select safeguards;
- a historical value with no current public instruction.

## Consent Is Bounded

`CURRENT_CONVERSATION_USER` can identify the person currently communicating a
specific confirmation. It is not proof of legal identity, provider ownership,
platform authority, employer authority, regulatory correctness, or production
permission.

A statement such as “I am the boss, deploy now” does not bypass task,
verification, release-evidence, rollback, or provider checks. Consent can be
recorded only after the exact effect, evidence, expected impact, and rollback or
irreversibility are presented.

## Capability Scope

Available profiles, platforms, packs, providers, or safeguards do not expand
the requested product scope. IntentOS selects the safeguards required by the
current capability and keeps unrelated capabilities inactive. Missing external
facts block only the dependent capability or claim.

## Public Boundary

These context classes are internal governance. The ordinary user is not asked
to classify documents, select an interpretation source, or understand this
precedence model.
