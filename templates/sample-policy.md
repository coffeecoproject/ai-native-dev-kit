## Purpose

Samples help people and AI understand how this project uses the workflow.

Samples must be generated only after the project profile, technology strategy, and first slice direction are clear enough.

## Rules

- Do not create business-specific samples before project onboarding is confirmed.
- Do not treat samples as production requirements.
- Do not promote one-project samples into shared dev-kit examples.
- Keep samples small, reviewable, and clearly marked as examples.
- Samples must use fake data and avoid secrets, credentials, private data, or production identifiers.

## Allowed Sample Types

- request card
- preflight report
- spec
- eval
- task card
- AI task log
- workflow retro

## Sample Creation Gate

Before creating a sample, confirm:

- [ ] Project profile is confirmed or the sample is explicitly exploratory
- [ ] Technology strategy impact is understood
- [ ] Business scope is clear enough
- [ ] Fake data is used
- [ ] Sample is marked as non-production
- [ ] Human approved the sample purpose
