## Human Summary

The first read-only calibration batch shows the baseline selector is safe by
default: it avoided target writes, kept BL2 as candidate-only, and protected
production and dirty-worktree projects. The main gaps are precision issues, not
boundary failures.

## What Worked

1. Production governed projects stayed adapter-only and read-only.
2. Dirty worktree projects stopped write actions.
3. BL2 was treated as a candidate path, not an automatic install.
4. Existing Web engineering and environment baselines were detected.
5. Mini Program and payment risk were detected.
6. A large iOS-led governed monorepo stayed read-only and adapter-first.

## What Needs Calibration

1. Mini Program cloud functions should count as a backend companion signal.
2. Dirty-worktree output should still show "possible BL2 candidate after
   worktree resolution" when high-risk signals exist.
3. Internal-admin detection should require stronger evidence than permission or
   backend vocabulary.
4. Industrial candidate lists should be visually separated from selected packs.
5. User-facing recommendations should use clearer language for existing
   governed projects: map first, do not install a parallel workflow.
6. Legacy `baseline` and guided baseline output should explain BL1 safe action
   versus BL2 candidate target level without sounding contradictory.
7. Monorepo platform output should distinguish selected, present but inactive,
   present but needs confirmation, and not detected.

## Recommended Next Implementation Order

1. Calibrate detection language and candidate labeling only.
2. Add fixture coverage for the three project shapes in this batch.
3. Keep all target-project write behavior unchanged.
4. Do not add new industrial packs until selection precision is improved.
